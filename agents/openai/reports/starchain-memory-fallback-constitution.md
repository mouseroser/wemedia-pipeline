# 星链打磨层 - Step 1.5C：OpenAI 宪法

> **主题**：`memory-lancedb-pro` 的 Layer 3（NotebookLM）降级集成宪法  
> **目的**：在不破坏主链稳定性的前提下，为 Layer 2 召回增加可控、可降级、可观测的深度补充能力。

---

## 一、核心约束（必须遵守）

### 1. 主链稳定性优先于深度能力
- Layer 3 只是 **降级增强**，不是主链依赖。
- 任何情况下，`memory_recall` 都必须保证：**Layer 3 失败 ≠ 主流程失败**。
- 若 Layer 3 超时、429、解析失败、服务不可用，必须自动回退为 **仅返回 Layer 2 结果**。

**可验证标准**：人为制造 NotebookLM 失败时，`memory_recall` 仍返回有效结果，且主 Agent 不崩溃。

### 2. 触发必须可解释、可配置、可收敛
- Layer 3 触发只能由 **明确规则** 决定，不允许隐式或不可追踪的“黑箱判断”。
- 触发依据仅限两类：
  1. **查询特征**：时间敏感、推理敏感、显式深度要求；
  2. **Layer 2 质量不足**：结果数过少、top1 分过低、top3 平均分过低。
- 所有阈值与关键词必须来自配置，可调整、可回滚。

**可验证标准**：每次触发都能记录 reasons / metrics / config snapshot，并可复现。

### 3. 失败边界必须前置收口
- Layer 3 的 timeout 必须被严格限制在 **不会先拖死主 Agent** 的范围内。
- 当前阶段默认采用 **同步调用 + 短超时 + 失败回落**，禁止以“深度更重要”为由放宽超时边界。
- 在没有异步架构前，不得让 Layer 3 成为长阻塞路径。

**可验证标准**：高延迟场景下，Layer 3 先超时并回退，而不是主流程先超时。

### 4. 集成形态以最小改动、最小依赖为原则
- 当前阶段允许保留 CLI spawn 方案，因为它实现简单、隔离清晰、便于上线。
- 在未验证 SDK 稳定性前，不强行切换为更“优雅”但依赖更重的方案。
- 架构演进顺序必须是：**先稳定，再观测，再重构**。

**可验证标准**：首版不引入额外复杂基础设施；改动集中在现有拦截点与配置层。

### 5. 可观测性不是附属品，而是交付的一部分
- 每次 Layer 3 触发、成功、失败、超时、重试，都必须有结构化记录。
- 没有监控数据的“成功”不算成功；没有失败统计的“稳定”不算稳定。

**可验证标准**：能统计 fallback rate、success rate、timeout rate、平均耗时、主要触发原因。

## 二、实现边界（范围界定）

### 必须做
1. 保留并使用现有拦截点：`registerMemoryRecallTool -> execute()`。
2. 保留 Layer 2 作为主链，Layer 3 作为条件性补充。
3. 为 Layer 3 设置保守超时（原则上不高于主 Agent 安全窗口）。
4. 确保 Layer 3 失败时自动降级到 Layer 2 only。
5. 输出触发原因、关键评分指标、失败原因。
6. 对 429 / 限流类错误加入有限重试与退避。

### 明确不能做
1. **不能**把 Layer 3 设计成必经路径。
2. **不能**为了追求完整答案而牺牲主链响应稳定性。
3. **不能**把 NotebookLM 查询结果当作覆盖 Layer 2 的唯一真相源。
4. **不能**在首版中引入重型异步队列、复杂任务编排、跨组件大改造。
5. **不能**接受“偶发主流程超时但答案更完整”的取舍。

### 可以延后
1. `sessions_spawn` / SDK 直连方案替换。
2. 完整异步投递模式（后台任务 + 私信推送）。
3. 流式返回 / onUpdate 类体验优化。
4. 更精细的熔断器、任务队列、持久化统计面板。
5. 单独的 `memory_recall_deep` 工具拆分。

## 三、验收标准（必须可测试）

### A. 功能正确性
- 当查询命中配置关键词时，能正确触发 Layer 3。
- 当 Layer 2 结果数或分数低于阈值时，能正确触发 Layer 3。
- 当未命中触发条件时，不应无谓调用 Layer 3。

### B. 稳定性
- NotebookLM 不可用时，`memory_recall` 仍返回 Layer 2 内容。
- Layer 3 超时时，工具返回明确降级结果，而非主流程报错。
- 连续出现 429 / rate limit 时，系统表现为有限重试后回退，而不是无限等待或级联失败。

### C. 性能边界
- 普通未触发查询应维持近似现有 Layer 2 响应时间。
- 触发 Layer 3 的查询，其最坏行为应是“在受控时间内回退”，而不是无限阻塞。
- 不允许因少量深度查询导致整体 recall 工具可用性显著下降。

### D. 可观测性
- 每次调用都能在 `details` 或日志中看到：
  - 是否触发 Layer 3
  - 触发原因
  - 结果数 / top1 / avgTop3
  - Layer 3 耗时
  - 成功/失败/超时状态
- 能够基于运行数据回答：
  - fallback 触发率是多少？
  - 失败主要因为什么？
  - timeout 是否高于预期？

### E. 可回滚性
- 关闭 `layer3Fallback.enabled` 后，系统应恢复为纯 Layer 2 模式。
- 调整关键词、阈值、timeout 不应要求改代码主干。

## 四、风险红线（不可触碰）

### 红线 1：主流程崩溃
若 Layer 3 导致主 Agent/tool 超时、流水线中断、上层调用崩溃，则视为 **不可接受失败**。

### 红线 2：深度能力反客为主
若实现让 Layer 3 从“条件性补充”变成“实际主路径”，则违背架构目标，必须回退。

### 红线 3：不可解释触发
若系统无法说明“为什么触发 / 为什么没触发 / 为什么失败”，则不允许进入生产依赖场景。

### 红线 4：静默失败
若 Layer 3 失败但日志、details、监控中无痕迹，属于不可接受设计缺陷。

### 红线 5：过早复杂化
若首版为解决局部问题而引入大规模异步基础设施、跨组件重构、难回滚依赖，属于架构越界。

---

## 五、执行裁决（本阶段结论）

### 当前推荐路线
- **采纳方向**：以 **方案A（CLI spawn）** 作为当前交付基线。
- **强制补丁**：
  1. 收紧 timeout，确保失败边界先于主链超时；
  2. 增加有限重试与退避；
  3. 补齐结构化监控字段；
  4. 保证 Layer 3 失败只影响增强内容，不影响主结果。

### 后续演进顺序
- **v1**：同步降级版，先把稳定性与可观测性做对。
- **v1.x**：按数据调优触发阈值，必要时拆出 `memory_recall_deep`。
- **v2**：若深度查询频率和价值已被证实，再进入异步投递架构。

---

**一句话宪法**：  
> Layer 3 可以慢、可以失败、可以延后，但绝不能拖垮 Layer 2 主链，绝不能把“增强能力”做成“稳定性债务”。
