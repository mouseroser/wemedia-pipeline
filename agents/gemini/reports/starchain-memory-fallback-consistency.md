## 二、 发现的矛盾点

1. **超时边界的“数学矛盾”**
   - **宪法约束**：Layer 3 超时必须严格限制在不会拖死主 Agent 的范围内（宪法三）。
   - **Plan 设定**：Phase 1 将 timeout 从 90s 降至 60s。
   - **矛盾点**：主 Agent 的 Tool execution limit 通常正是 60s。Layer 2 检索（~0.5s） + Layer 3 超时（60s） + 进程冷启动/网络延迟，总耗时必定 `> 60s`，这依然会触发主流程崩溃红线。

2. **重试机制与超时的冲突**
   - **Plan 设定**：Phase 1.2 引入了针对 429 错误的重试机制，包含指数退避（等待 2s, 4s）。
   - **矛盾点**：在同步阻塞模式下，原执行耗时加上退避等待时间，以及重试的再次执行时间，极容易击穿本就紧张的超时窗口。

## 三、 遗漏的风险或场景

1. **上下文爆炸风险（Token 溢出）**
   - **场景**：NotebookLM 作为深度研究 Agent，经常返回长篇大论。如果触发 Layer 3 并在主输出中直接拼接其完整结果，极易导致调用方 Agent（如主干模型的 memory_recall）遭遇上下文长度截断或 Token 溢出。
   - **遗漏**：Plan 中未提及对 Layer 3 返回内容的长度限制或压缩机制。

2. **JSON 解析脆弱性的实操缺失**
   - **场景**：Scan 和 Research 均指出了 CLI spawn 方案中 JSON 解析脆弱的风险（可能混入控制字符或非 JSON 输出）。
   - **遗漏**：Plan 中将此问题的解决推迟或仅作为"降级兜底"（原样返回），但未在 Phase 1 安排低成本的防御措施（如正则提取包裹的 JSON），长文本兜底可能再次引发上下文污染。

## 四、 改进建议（针对 Plan 的执行前修正）

1. **调整超时安全阈值（必须执行）**
   - 将 Phase 1.1 中的默认 `timeout` 从 60s 进一步收紧至 **45s 或 50s**。为 Layer 2、进程开销及网络延迟保留至少 10-15s 的安全缓冲。

2. **为重试机制增加时间感知（必须执行）**
   - Phase 1.2 的重试逻辑中，必须加入对**全局剩余时间**的判断。如果第一轮失败后，剩余时间已不足以支持退避及第二轮安全执行，则直接熔断回退，不再重试。

3. **增加结果截断防御（建议执行）**
   - 在 Phase 1 错误降级逻辑中，增加对 `layer3Result.text` 的最大长度限制（如截断至 2000 或 3000 字符），并在末尾标注 `[截断...]`，防止单一深度查询挤爆主链上下文。

4. **前置轻量级 JSON 正则保护（建议执行）**
   - 在 CLI 调用结果解析处，不要直接 `JSON.parse(trimmed)`，而是使用正则（如 `/\{[\s\S]*\}/`）提取其中的 JSON 块再解析，将这个微小改动并入 Phase 1，提升解析鲁棒性。
